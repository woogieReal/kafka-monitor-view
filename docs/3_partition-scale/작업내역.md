## [2026-05-14] 파티션 증가 및 토픽 초기화 구조 개선

### 작업 내용
- `system-metrics` 토픽 파티션 1개 → 3개, 레플리카 3개로 변경
- `consumer/main.py`에 `group_id` 추가 및 메시지 로그에 `partition` 번호 출력
- `KAFKA_CREATE_TOPICS` 환경변수 제거 후 `kafka-init` 컨테이너로 토픽 생성 책임 이관
- 토픽 생성 스크립트를 `kafka-init/init-topics.sh`로 분리
- 브로커 3개 모두에 `KAFKA_AUTO_CREATE_TOPICS_ENABLE: "false"` 추가
- producer/consumer의 `depends_on`을 `kafka-init`의 `service_completed_successfully` 조건으로 변경
- README에 재시작 명령어 추가

### 주요 결정 사항
- **`kafka-init` 컨테이너 도입**: `KAFKA_CREATE_TOPICS`는 브로커 기동 시점에 실행되는데, 이 때 다른 브로커가 아직 클러스터에 합류하지 않아 replication-factor=3 설정이 적용되지 않는 타이밍 문제가 있었음. 모든 브로커가 준비된 이후 토픽을 생성하는 전용 컨테이너로 분리하여 근본 해결
- **`KAFKA_AUTO_CREATE_TOPICS_ENABLE: "false"` 적용**: kafka-init이 토픽을 생성하기 전에 producer가 먼저 auto-create하는 경쟁 조건을 차단하기 위해 브로커 레벨에서 자동 생성을 비활성화
- **`service_completed_successfully` 조건**: producer/consumer가 kafka-init 완료를 기다린 후 기동하도록 보장. 토픽이 없는 상태에서 producer가 연결 시도하는 문제 방지
- **스크립트 파일 분리**: `docker-compose.yml`의 `entrypoint` 인라인 스크립트보다 별도 파일이 가독성과 유지보수 면에서 유리

### 이슈 & 해결
- **YAML folded block(`>`)으로 인한 인수 분리 오류**: `entrypoint`에서 `>` 방식으로 멀티라인 명령을 작성하자 `--topic`, `--partitions` 등이 별도 명령으로 해석되어 실패. 배열 형식 + `|` (literal block)으로 변경하여 해결
- **producer와 kafka-init 경쟁 조건**: entrypoint 형식을 수정했음에도 토픽이 파티션 1개로 생성되는 문제 재발. 원인은 producer가 브로커 기동과 동시에 시작되어 `auto.create.topics.enable` 기본값(`true`)으로 토픽을 먼저 생성, kafka-init의 `--if-not-exists`가 이를 건너뛴 것. `KAFKA_AUTO_CREATE_TOPICS_ENABLE: "false"` + `depends_on` 조건을 함께 적용하여 해결

### 변경 파일
- `docker-compose.yml` — `KAFKA_CREATE_TOPICS` 제거, `KAFKA_AUTO_CREATE_TOPICS_ENABLE` 추가, `kafka-init` 서비스 추가, producer/consumer depends_on 변경
- `consumer/main.py` — `group_id` 추가, 로그에 `partition` 번호 출력
- `kafka-init/init-topics.sh` — 브로커 준비 대기 및 토픽 생성 스크립트 (신규)
- `README.md` — 재시작 명령어 추가
