# 작업중 이슈

## KAFKA_CREATE_TOPICS 타이밍 문제

### 증상

`docker-compose down && up` 이후 `system-metrics` 토픽이 파티션 1개, 레플리카 1개로 잘못 생성됨.

### 원인

`kafka1` 시작 시 `KAFKA_CREATE_TOPICS` 환경변수로 토픽을 생성하는데, 이 시점에 `kafka2`, `kafka3`가 아직 클러스터에 합류하지 않은 상태임.

```
kafka1 기동 → KAFKA_CREATE_TOPICS 실행 (브로커 1개만 존재)
             → replication-factor=3 요청 실패
             → producer가 먼저 auto-create (기본값: 파티션 1, 레플리카 1)
```

### 현재 상태

`kafka-topics.sh --alter`로 파티션을 3개로 직접 수정하여 임시 해결한 상태.
다음 `docker-compose down && up` 시 동일 문제 재발함.

### 해결 방안

**방안 1: init 컨테이너 도입 (권장)**
- `KAFKA_CREATE_TOPICS` 제거
- 모든 브로커가 준비된 이후 토픽을 생성하는 별도 컨테이너 추가
- 타이밍 문제를 근본적으로 해결

**방안 2: healthcheck + depends_on 조건 추가**
- `kafka1`, `kafka2`, `kafka3`에 healthcheck 설정
- 모든 브로커 healthy 확인 후 토픽 생성 순서 보장

---

## 해결 진행 내역 (방안 1 적용)

### 시도 1 — entrypoint 줄바꿈 문제

`docker-compose.yml`의 `entrypoint`에서 YAML `>` (folded block) 방식으로 멀티라인 명령을 작성했더니, `kafka-topics.sh` 인수들이 각각 별도 명령으로 해석되어 실패.

```
/bin/bash: line 8: --topic: command not found
```

→ `entrypoint` 배열 형식 + `|` (literal block)으로 변경하여 해결.

### 시도 2 — producer와 kafka-init 간 경쟁 조건

entrypoint 형식을 고쳤음에도 토픽이 파티션 1개로 생성되는 문제가 재발.

원인: producer도 브로커가 뜨는 즉시 시작되어 `auto.create.topics.enable` 기본값(`true`)으로 토픽을 먼저 생성. kafka-init의 `--if-not-exists`가 이미 존재하는 토픽을 건너뜀.

→ 두 가지를 함께 적용하여 해결:
1. 모든 브로커에 `KAFKA_AUTO_CREATE_TOPICS_ENABLE: "false"` 추가 — producer의 자동 토픽 생성 차단
2. producer/consumer의 `depends_on`에 `condition: service_completed_successfully` 추가 — kafka-init이 정상 종료된 이후에만 기동

### 최종 결과

```
kafka-init: PartitionCount: 3  ReplicationFactor: 3
  Partition 0 → Leader: kafka3, Replicas: 3,1,2
  Partition 1 → Leader: kafka1, Replicas: 1,2,3
  Partition 2 → Leader: kafka2, Replicas: 2,3,1
```

consumer 로그에서 `partition=0`, `partition=1`, `partition=2` 모두 수신 확인.
